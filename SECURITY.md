# Security Policy

## Reporting a Vulnerability

If you discover a security vulnerability in Pinky, please report it responsibly.

**Do not open a public issue.** Instead, email **security@pinky.dev** with:

1. Description of the vulnerability
2. Steps to reproduce
3. Impact assessment
4. Suggested fix (if any)

We will acknowledge your report within **48 hours** and aim to provide a fix within **7 days** for critical issues.

## Security Architecture

### Encryption

- **Credentials at rest:** AES-256-GCM envelope encryption with key versioning and Additional Authenticated Data (AAD) binding. Cluster tokens are encrypted before storage and decrypted only at the point of use.
- **Encryption key:** Set via `PINKY_ENCRYPTION_KEY` environment variable (64-character hex string = 256-bit key). Auto-generated by Helm chart and preserved across upgrades.

### Identity Isolation

- **Observer identity:** A cluster-scoped service account with read-only RBAC used for scanning. Never used for writes or sensitive reads.
- **User identity:** Per-user cluster bindings via OAuth token. Remediation actions (patch, scale, delete) always use the approving user's token, never the observer SA.
- **Separation enforced:** `apply_change` requires a `binding_id` — no fallback to observer credentials.

### Credential Redaction

Evidence gathered from clusters is redacted before being sent to LLM providers. 11 redaction patterns cover:
- Bearer tokens, Basic auth headers
- Connection strings (postgres://, redis://, mongodb://)
- Environment variables containing keys/secrets/passwords
- Kubernetes service account JWTs
- Kubeconfig credential fields
- AWS/GCP metadata tokens
- Helm release secrets

### Authentication & Authorization

- **Session management:** HTTP-only, Secure, SameSite=Strict cookies with configurable idle/absolute timeouts. Sessions stored in Redis.
- **API tokens:** HMAC-SHA256 hashed, plaintext returned once on creation. Bearer auth alongside session cookies.
- **Authorization model:** Three layers — product authz (admin/user), cluster authz (observer read/user read/user write), execution authz (risk class, approval status, session age).
- **CSRF protection:** Double-submit cookie pattern with `X-CSRF-Token` header.

### Container Security

- All containers run as non-root (UID 1001)
- Read-only root filesystem
- Strict Content Security Policy (no unsafe-inline, no unsafe-eval)
- Security headers middleware (X-Content-Type-Options, X-Frame-Options, etc.)

### TLS

- All K8s API calls verify TLS certificates using in-cluster CA or system CA bundle
- TLS verification is only disabled when `PINKY_DEBUG=true` (development clusters with self-signed certs)

## Secret Management

- Secrets are stored in a gitignored `secrets/` directory, never committed
- `scripts/deploy.sh` creates Kubernetes secrets from local files before Helm install
- Pre-commit hooks scan for accidental secret commits
- `.gitignore` explicitly excludes `.env`, `secrets/`, credential files
