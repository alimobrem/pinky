# Pinky SDS

## Overview

`Pinky` is a greenfield operations platform with a bundled UI, API, and worker runtime. Its embedded SRE agent, `The Brain`, investigates issues, prepares plans, and drives safe execution workflows. The platform is built around task-first operations, not raw signal accumulation. Every operation is instrumented for analytics and ROI measurement.

## Recommended Stack

- UI: Next.js + React + TypeScript
- API: FastAPI (async)
- Workflow runtime: Temporal
- Database: PostgreSQL (system of record + projections)
- Session store: Redis
- Real-time updates: SSE
- Encryption: AES-256-GCM envelope encryption
- Migrations: Alembic
- Shared schemas/contracts: typed package shared between UI and backend

## Runtime Model

Pinky is **async-first, worker-executed, and workflow-driven**.

### What should be async-first
- FastAPI request handlers, SSE streaming, database access, workflow submission, cluster observation loops

### What should not run inline in the API path
- Long-running investigations, remediation execution, verification waits, CPU-heavy correlation, large topology work, blocking SDK/provider logic

### Design rule
If a task cannot complete quickly and deterministically within a normal request budget, it becomes a durable workflow step or worker-executed activity.

## Auth and Session Design

### Product auth flow
1. User lands on login screen, chooses OpenShift or external OIDC
2. Backend performs provider redirect and callback exchange
3. Backend resolves or creates the Pinky principal (auto-link on verified email match)
4. Backend issues secure product session (Redis + audit log to Postgres)
5. Backend redirects to same-origin app shell

### Session store
- **Redis** for active sessions: TTL-based expiry, atomic rotation/revocation
- **PostgreSQL** `session_audit_log` for compliance
- Session data encrypted with AES-256-GCM (ADR-03)
- Redis is ephemeral: loss = re-login

### Session model
- Idle timeout, absolute timeout, refresh window
- Rotation after login / privilege elevation
- Logout and revocation, CSRF token (double-submit cookie, `X-CSRF-Token` header)
- If session expires during SSE stream: stream terminates with `auth-expired` event

### Cluster access model
- Observer bindings for standard non-sensitive reads
- User-scoped bindings for sensitive reads and writes
- Binding states: `missing`, `valid`, `expiring`, `expired`, `revoked`
- **>5 clusters requires external OIDC** — per-cluster OAuth only for ≤5 clusters
- Backend forwards observer credentials for standard reads, user credentials for sensitive reads/writes
- `401` = session/binding freshness lost; `403` = user lacks permission

## Authorization Model

Three layers: Product authz, Cluster authz, Execution authz.

Execution authz classes:
- product-authenticated read
- observer read
- user-sensitive read (logs, YAML, secrets, exec, private app data)
- cluster-user write (remediation, rollback, state-changing actions)
- admin-only control-plane action

## Security Baseline

### Browser security
Strict CSP (no unsafe-inline, no unsafe-eval), same-origin, HSTS, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.

### Secret handling
AES-256-GCM envelope encryption for tokens/bindings. KMS in production, env-var key in dev. 90-day key rotation. Evidence bundles redacted before prompts. Redaction failures fail closed.

## Observation and Issue Processing

### Observer loop
One async worker per cluster, polling at 60s default. Max 50 concurrent observers. Backpressure: skip scan cycle when queue depth >500 per cluster.

### Correlation key algorithm

```python
def compute_correlation_key(cluster_id, resource_kind, resource_namespace, resource_name, scanner, check_id):
    raw = f"{cluster_id}:{resource_kind}:{resource_namespace}:{resource_name}:{scanner}:{check_id}"
    return hashlib.sha256(raw.encode()).hexdigest()
```

### Dedup rules
1. Same fingerprint for same cluster+scanner: upsert observation
2. Same correlation key as open issue: attach to existing issue
3. Same correlation key as recently resolved issue (<1hr): reopen instead of creating new
4. Identical fingerprint within scan interval: collapse to latest

### Investigation trigger conditions
All must be true: issue open >2 consecutive scan cycles, severity >= policy threshold, no valid recent investigation artifact with matching evidence hash, issue not suppressed.

## SSE Protocol

### Wire format
Standard `text/event-stream`. Each event: `id: <event_id>\nevent: <type>\ndata: <JSON>\n\n`

### Heartbeat
15-second interval. Client treats no event for 45s as dead connection.

### Reconnection
Client sends `Last-Event-ID`. Server replays from buffer (1000 events per stream, LRU eviction). If outside buffer: `snapshot_required` event with `fetch_url`. Backoff: 1s → 2s → 4s → 8s → 30s cap. Reset to 1s after 60s connected.

### Stream endpoints

| Endpoint | Purpose |
|---|---|
| `/v1/streams/executions/{id}` | Live execution progress |
| `/v1/streams/work-items` | Work item status changes |
| `/v1/streams/watch` | Watch/analysis activity |
| `/v1/streams/issues` | Issue lifecycle changes |

### SSE fan-out
API workers listen to PostgreSQL `NOTIFY` on execution_events/work_items tables. In-memory SSE connection registry per pod. Missed events backfilled from Postgres on reconnect. No shared state between API pods for SSE.

## Temporal Workflow Patterns

Namespace: `pinky`. Task queues: `investigation`, `remediation`, `observation`, `projection`.

### InvestigationWorkflow
```python
class InvestigationInput(BaseModel):
    issue_id: UUID
    cluster_id: UUID
    correlation_key: str
    evidence_hash: str

class InvestigationResult(BaseModel):
    artifact_id: UUID
    summary: str
    recommended_action: str
    confidence: float
    tool_calls: list[str]
```
Flow: gather evidence (30s) -> check artifact cache (5s) -> LLM investigation (120s, heartbeat 30s) -> store artifact (5s). Workflow ID: `investigation-{cluster_id}-{fingerprint_hash}` for dedup.

### ApprovalWorkflow
Emits `approval_required`, waits for signal or 4hr timeout. Signals: `approve(payload)`, `reject(payload)`. Returns `approved`, `rejected`, or `expired`.

### RemediationWorkflow
Validates approval -> executes steps using user binding (60s per step) -> triggers VerificationWorkflow as child.

### VerificationWorkflow
Waits configurable delay (default 60s) -> re-scans resources -> compares against expected state -> emits `verified` or `failed`.

## Projection and Reconciliation

### Pipeline
```
Temporal Event History → Projector Worker (2s poll) → Event Transform → Idempotent Upsert → Postgres → NOTIFY → SSE
```

- Projector tracks cursors in `projection_cursors` table
- Writes are `INSERT ... ON CONFLICT (execution_id, sequence) DO NOTHING`
- Lag metric: `projection_lag_seconds = now() - max(occurred_at)`. Alert threshold: >10s.

## Definition System

### Loading strategy
1. On startup, load all `.md` files from `definitions/` directory tree
2. Parse frontmatter (YAML) and body (markdown)
3. Register in-memory with kind + name as key
4. Load DB `definitions` table entries
5. DB entries override filesystem entries for same `(kind, name)`
6. Cache with configurable TTL (~30s) for runtime additions via API

### Tool credential resolution

Tools declare auth needs in frontmatter. Runtime resolves at execution time:

```
Tool frontmatter: authz_class or service
         ↓
Execution context: principal_id, cluster_id, execution_id
         ↓
Runtime resolves:
  Cluster tool (authz_class): principal → cluster_identity_binding → decrypt → K8s client
  Observer tool (authz_class: observer_read): cluster → cluster_observer_binding → decrypt → K8s client
  External service tool (service): service name → service_binding → decrypt → HTTP client
         ↓
Authenticated client injected into tool executor
         ↓
Tool never touches raw credentials
```

Safety: observer tools cannot escalate to user-write. External service credentials never in prompts/logs. Resolution failure = explicit error, not silent fallback.

### Policy rule engine

Declarative rules evaluated deterministically (no LLM):
1. Load enabled rules from `policy_rules` table + `definitions/policies/` directory, ordered by priority
2. For each new/updated issue, iterate rules
3. First matching rule wins (short-circuit)
4. If no rule matches, fall through to configurable default (observe or investigate)
5. Every decision logged to `analytics_events`

## Timeout and Resilience Model

### Session/auth
| Parameter | Value |
|---|---|
| Idle timeout | 30 min |
| Absolute timeout | 8 hours |
| Reauth threshold (sensitive) | 15 min |
| Binding expiry propagation | immediate |

### LLM/execution
| Parameter | Value |
|---|---|
| Utility model timeout | 15s |
| Reasoning model timeout | 120s |
| Synthesis model timeout | 60s |
| Interactive model timeout | 30s |
| Max LLM retries | 3 with exponential backoff |

### Workflow/runtime
| Parameter | Value |
|---|---|
| Approval expiry | 4 hours (configurable per risk class) |
| Verification delay | 30s–5min (configurable) |
| Job lease timeout | 10 min |
| Workspace hydration timeout | 10s, fallback to cached artifact |

### Failure modes
| Failure | Degradation |
|---|---|
| Redis unavailable | Fall back to Postgres session lookup. New sessions blocked. Alert. |
| Postgres unavailable | API returns 503. SSE sends error event. Workflows pause. |
| Temporal unavailable | New submissions queued in-memory (bounded 100). API returns 503 for execution-start. |
| LLM provider timeout | Return cached investigation if <1h old. Otherwise mark work item `blocked`. |
| Cluster unreachable | Observer marks `degraded`. After 3 consecutive failures, mark `unhealthy`. |
| SSE connection drop | Client reconnects with backoff. Server replays from buffer. Outside buffer: `snapshot_required`. |

## Analytics and ROI

### Analytics events
Every significant action is logged to `analytics_events`:
- Auth: login, logout, session rotation, binding create/refresh/revoke
- Policy: signal received, policy decision (suppress/observe/investigate/auto-resolve/create-task)
- User actions: task accept/start/complete/reassign, approval grant/reject, page views
- Brain actions: investigation started/completed, plan generated, remediation executed, verification result
- Outcomes: verified_fixed, recurred, operator_overridden, rolled_back, dismissed

### ROI metrics (queryable via `/v1/analytics/roi`)
- Time from signal to task creation
- MTTR with Pinky vs baseline
- Issues resolved (auto + assisted)
- Automation success rate (% passing verification)
- Cost per resolution (tokens / resolved issues)
- Operator override rate
- Confidence calibration (predicted vs actual)
- Recurrence rate (% reopened within 7 days)

### Scanner quality metrics (queryable via `/v1/analytics/scanners`)
- Signal volume per scanner (total, suppressed, escalated, tasked)
- False positive rate per scanner
- Noise ratio per scanner
- Scanner ROI: issues-resolved-per-cost by source

## Deployment Model

| Component | Type | Replicas | Notes |
|---|---|---|---|
| `web` | Stateless | 2+ | Next.js, behind ingress |
| `api` | Stateless | 2+ | FastAPI, behind ingress |
| `worker` | Stateless | 2+ | Temporal workers, observers, projectors |
| `postgres` | Stateful | 1 primary + 1 replica | PVC-backed, operator-managed |
| `redis` | Stateful | 1 primary + 1 replica | Session cache, SSE pub/sub |
| `temporal-server` | Stateful | 1 (dev) / 3 (prod) | Own Postgres backend |

### Ingress routing (same-origin)
```
pinky.example.com/          → web
pinky.example.com/api/v1/   → api
pinky.example.com/api/v1/streams/ → api (longer timeouts, no buffering)
```
